# Plan: `apply.py bundle` + tags

## Context

The dotfiles repo has two needs: (1) separate home vs work content so they don't cross-contaminate, and (2) deploy configs to ephemeral environments without git clone. The solution is a tag system (inspired by rcm's `tag-*` directories) for scoping which files are active, plus an `apply.py bundle` command that generates a self-contained shell script for credential-free deployment.

Tags apply to both `homelinks*/` (symlinked files) and `include*/` (sourced files).

## Tag system

### Directory layout

```
dotfiles/
├── homelinks/                # always linked (base)
├── homelinks-<tag>/          # linked when <tag> active
├── include/                  # always sourced (base)
│   ├── aliases.sh
│   └── fn/
│       └── *.sh
├── include-<tag>/            # sourced when <tag> active
│   └── ...
├── include.d/                # GENERATED by apply.py — not checked into git
│   ├── aliases.sh            # -> ../include/aliases.sh
│   ├── fn/
│   │   └── ...               # symlinks to files from include/ + include-<tag>/
│   └── ...
├── customs/                  # symlink to all_customs/<active machine>
│                             # created/updated by `apply.py apply`
│                             # other files (e.g. gitconfig) reference this path
├── all_customs/              # one subdir per machine
│   ├── my-laptop/
│   │   ├── machinename
│   │   ├── tags              # one tag per line
│   │   ├── path_overrides
│   │   └── gitconfig
│   └── ...
```

### `all_customs/<name>/tags` format

Plain text, one tag per line (rcm-style):

```
personal
go
```

### `include.d/` — generated symlink directory

`apply.py` reads the machine's tags and builds `include.d/` with symlinks to files from `include/` + each active `include-<tag>/`. This is a flat merge — the directory structure inside each source is preserved, but the tag prefix is stripped:

- `include/fn/common.sh` → `include.d/fn/common.sh`
- `include-personal/fn/personal.sh` → `include.d/fn/personal.sh`
- `include-go/paths.sh` → `include.d/paths.sh`
- `include-work/paths.sh` → name collision with above

**Name collisions:** If two tags provide the same relative path (e.g. `include-go/paths.sh` and `include-work/paths.sh`), `apply.py` should error and tell the user. This keeps things simple — rename one of them to be unique. In practice this is rare.

`include.d/` is gitignored and rebuilt by `apply.py` on every run. The command is:

```
python3 apply.py build-includes --customs <name> [--dry-run]
```

`--dry-run` prints which symlinks would be created (and detects collisions) without writing anything. Without `--dry-run`, it still prints before executing.

This is also run automatically as part of `apply.py apply`.

```python
def collect_include_entries(config: MachineConfig) -> dict[Path, Path]:
    """Collect include files: {relpath -> source_path}. Raises on conflicts."""
    tags = config.tags()
    entries: dict[Path, tuple[Path, str]] = {}
    source_dirs = [config.top / "include"] + [config.top / f"include-{t}" for t in tags]
    for source_dir in source_dirs:
        if not source_dir.is_dir():
            continue
        for f in source_dir.rglob("*"):
            if f.is_file():
                relpath = f.relative_to(source_dir)
                if relpath in entries:
                    _, prev_dir = entries[relpath]
                    raise Exception(
                        f"Conflict: {relpath} provided by both "
                        f"{prev_dir} and {source_dir.name}"
                    )
                entries[relpath] = (f, source_dir.name)
    return {relpath: source for relpath, (source, _) in entries.items()}


def build_include_d(config: MachineConfig, dry_run: bool = False) -> None:
    """Rebuild include.d/ from include/ + include-<tag>/ for active tags."""
    include_d = config.top / "include.d"
    entries = collect_include_entries(config)

    for relpath, source in sorted(entries.items()):
        print(f"  {source} -> include.d/{relpath}")

    if dry_run:
        return

    # Wipe and recreate
    if include_d.exists():
        shutil.rmtree(include_d)

    # Create symlinks
    for relpath, source in entries.items():
        dest = include_d / relpath
        dest.parent.mkdir(parents=True, exist_ok=True)
        dest.symlink_to(source.absolute())
```

### Shell sourcing (in .zshrc and .bashrc)

Shell configs know nothing about tags. They just source from `include.d/`:

```sh
# Source all includes
. $HOME/dotfiles/include.d/aliases.sh
for f in $HOME/dotfiles/include.d/fn/*.sh; do source "$f"; done
```

The only change from today's .zshrc is `include/` → `include.d/`.

### `MachineConfig` — replaces `Paths`

`Paths` is renamed to `MachineConfig` to represent a repo + specific machine configuration. Tags become a method on the config rather than a standalone function.

```python
class MachineConfig:
    def __init__(self, top=None, dest=None, customs_name: str | None = None):
        ...
        self.customs_name = customs_name

    def homelinks(self) -> Path:
        return self.top / "homelinks"

    def customs_dir(self) -> Path | None:
        if self.customs_name is not None:
            return self.top / "all_customs" / self.customs_name
        # Fallback: customs/ symlink (backward compat)
        customs_path = self.top / "customs"
        if customs_path.exists():
            return customs_path
        return None

    def tags(self) -> list[str]:
        cd = self.customs_dir()
        if cd is None:
            return []
        tags_file = cd / "tags"
        if not tags_file.exists():
            return []
        return [line.strip() for line in open(tags_file) if line.strip()]

    def path_overrides(self) -> dict:
        base = read_path_overrides(self.top / "path_overrides")
        cd = self.customs_dir()
        custom = read_path_overrides(cd / "path_overrides") if cd else {}
        return base | custom
```

```python
def get_all_dotfile_entries(config: MachineConfig) -> list[DotfileEntry]:
    """Collect from homelinks/ + homelinks-<tag>/ for each active tag."""
    entries = get_dotfile_entries_from(config, config.homelinks())
    for tag in config.tags():
        tag_homelinks = config.top / f"homelinks-{tag}"
        if tag_homelinks.is_dir():
            entries += get_dotfile_entries_from(config, tag_homelinks)
    return entries
```

`plan_links()` and `apply_links()` use `get_all_dotfile_entries()`.

### No conflict resolution needed

Tags only add files. If a tagged homelinks dir has a file that also exists in base `homelinks/`, `plan_links()` will see the base symlink already exists and show it as a `replace` — the user can inspect with `--dry-run` first. For includes, collisions are an error (see above).

## CLI redesign

The old interactive menu and per-file prompting is replaced by a simpler machine-centric interface. The user creates their `all_customs/<name>/` directory beforehand.

### New commands

```
apply.py [--top <dir>] apply --customs <name> [--dry-run]
apply.py [--top <dir>] build-includes --customs <name> [--dry-run]
apply.py [--top <dir>] bundle --customs <name> --output <file>
```

`--top` specifies the dotfiles repo directory (defaults to cwd). Useful for testing.

**`apply --customs <name>`**: Creates/updates the `customs/` symlink, rebuilds `include.d/`, then applies all homelink symlinks. All commands print what they'll do before executing. Use `--dry-run` to preview without making changes.

**`build-includes --customs <name> [--dry-run]`**: Rebuilds `include.d/` for the given machine's tags. Prints symlinks before creating them. With `--dry-run`, prints only.

### `customs/` symlink management

The `customs/` symlink is a real artifact that other config files depend on (e.g. gitconfig includes `$HOME/dotfiles/customs/gitconfig`). The `apply` command creates/updates it:

```python
def ensure_customs_symlink(config: MachineConfig) -> None:
    """Create/update customs/ symlink to point to all_customs/<name>/."""
    if config.customs_name is None:
        return
    customs_link = config.top / "customs"
    target = Path("all_customs") / config.customs_name
    if customs_link.is_symlink():
        if customs_link.readlink() == target:
            return
        customs_link.unlink()
    elif customs_link.exists():
        raise Exception(f"customs/ exists and is not a symlink")
    customs_link.symlink_to(target)
```

### `apply_links()` replaces `make_links_cli()`

```python
def apply_links(config: MachineConfig, dry_run: bool = False) -> None:
    """Apply all symlinks for the given machine config."""
    link_ops = LinkOperations(config)
    planned = link_ops.plan_links()

    for op in planned:
        if op.action == "noop":
            print(f"  ok: {op.dest_path}")
        elif op.action == "create":
            print(f"  create: {op.source_path} -> {op.dest_path}")
        elif op.action == "replace":
            print(f"  replace: {op.source_path} -> {op.dest_path}")

    if dry_run:
        return

    for op in planned:
        link_ops.execute_link(op)
```

## Bundle system

### Key design decisions

- `bundle.sh` is a plain shell script with embedded file contents (base64-encoded)
- It writes homelinks files to their resolved target paths (after applying path_overrides)
- It recreates a `~/dotfiles/` tree: `include.d/` (flat files, no symlinks) and `customs/` — since .zshrc references `$HOME/dotfiles/...`
- Bundle is customs+tags-specific: reads `all_customs/<name>/tags` to determine contents

### Data flow

```
apply.py bundle --customs <name> --output bundle.sh
  │
  ├─ Read all_customs/<name>/tags → list of active tags (via config.tags())
  ├─ Read all_customs/<name>/path_overrides → merged with base path_overrides
  ├─ Collect homelinks/ + homelinks-<tag>/ files via get_all_dotfile_entries()
  ├─ Collect include/ + include-<tag>/ files via collect_include_entries()
  ├─ Collect customs files (machinename, gitconfig, tags)
  │
  └─ Emit bundle.sh
```

### Interface

```python
def generate_bundle(config: MachineConfig) -> str:
    """Generate a self-contained shell script for deploying dotfiles."""
```

### Generated bundle.sh structure

The `install_file` function reads base64 content from a heredoc on stdin:

```sh
#!/bin/sh
# Generated by apply.py bundle
# Customs: <name>
# Tags: personal, go
set -e

install_file() {
  mkdir -p "$(dirname "$1")"
  base64 -d > "$1"
  chmod "$2" "$1"
}

# Homelinks
install_file "$HOME/.zshrc" 644 << 'ENDOFFILE'
<base64 content>
ENDOFFILE

# Includes
install_file "$HOME/dotfiles/include.d/aliases.sh" 644 << 'ENDOFFILE'
<base64 content>
ENDOFFILE

# Customs
install_file "$HOME/dotfiles/customs/machinename" 644 << 'ENDOFFILE'
<base64 content>
ENDOFFILE
```

Note: bundle writes include files directly to `include.d/` and customs files to `customs/` (plain files, not symlinks) — the shell doesn't care whether they're symlinks or real files.
